# Практическая инструкция: фильтр URL в SiteEdit (для начинающего)

## 1. Зачем это нужно

Этот фильтр помогает:

- убирать из индекса мусорные URL;
- склеивать дубли через 301;
- отдавать 403/404 там, где страницы быть не должно;
- держать правила отдельно для каждого сайта в мультисайте.

Примеры мусорных URL, которые фильтр закрывает:

- `/page/tel:7952.../`
- `/page/79276078748/`
- `/page/NOCLICK_/`
- `/page/[/USER&/`
- `/?utm_source=...`
- `/?search_source=...`
- `/?disableGlobalInfoCollect=false`
- `/?_ym_debug=1`
- `/?` (пустой query)

## 2. Где находится логика

Основной код фильтра:

- `system/main/classes/seData.class.php`
- метод `filterRequestByRules()`
- метод `getUrlFilterRules()`

Дополнительно:

- `lib/cardimage.php` помечен `X-Robots-Tag: noindex` и отдаёт 404 без валидной `session`.

## 3. Куда класть файл правил в мультисайте

Для каждого сайта свой файл:

- основной путь: `projects/<SE_DIR>/urlfilter.json`
- запасной (совместимость): `projects/<SE_DIR>/cache/urlfilter.json`

Важно:

- `<SE_DIR>` — папка конкретного сайта в мультисайте;
- если есть файл в основном пути, fallback уже не используется.

## 4. Как работает фильтр (простыми словами)

### 4.1 Что делает 403

Для служебных путей (например, `/lib` и `/lib/...`) отдаётся `403 Forbidden`.

### 4.2 Что делает 404

Для мусорных хвостов вида `/slug/<мусор>/` отдаётся `404 Not Found`.

### 4.3 Что делает 301

Если URL технически валиден, но неканоничен, делается 301:

- `?item=1` или `?sheet=1` удаляются;
- трекинговые параметры удаляются (`utm_*`, `gclid`, `fbclid`, и др.);
- `/path/?` -> `/path/`.

## 5. Формат `urlfilter.json`

Нужны 4 массива:

- `forbidden_path_prefixes`
- `forbidden_alias_patterns`
- `query_drop_prefixes`
- `query_drop_keys`

Пример рабочего файла:

```json
{
  "forbidden_path_prefixes": [
    "/lib"
  ],
  "forbidden_alias_patterns": [
    "/^tel:/i",
    "/^noclick_$/i",
    "/^\\d{7,}$/",
    "/[\\[\\]&]/",
    "/[^a-z0-9\\-_]/i"
  ],
  "query_drop_prefixes": [
    "utm_"
  ],
  "query_drop_keys": [
    "gclid",
    "fbclid",
    "yclid",
    "msclkid",
    "wbraid",
    "gbraid",
    "search_source",
    "search_domain",
    "disableglobalinfocollect",
    "zarsrc",
    "ssp",
    "darkschemeovr",
    "setlang",
    "safesearch",
    "spm",
    "_ym_debug",
    "hl"
  ]
}
```

### 5.1 Подробно по каждому массиву

#### `forbidden_path_prefixes`

Что это:

- список префиксов пути, на которые нужно отдавать `403 Forbidden`.

Как проверяется в коде:

- берётся путь без домена и без query;
- путь приводится к виду относительно текущего сайта (`seMultiDir()`), чтобы не ломать мультисайт;
- каждый префикс сравнивается как отдельный сегмент: сработает для `/lib` и `/lib/...`, но не для `/library/...`.

Когда использовать:

- для служебных каталогов, которые не должны быть страницами (`/lib`, `/system` и т.п.).

Пример:

```json
"forbidden_path_prefixes": ["/lib", "/system"]
```

Важно:

- это именно `403`, а не `404`;
- используйте короткий корневой префикс (`/lib`), без regex.

#### `forbidden_alias_patterns`

Что это:

- список regex-шаблонов для мусорного alias в URL страницы.

Как проверяется в коде:

- правило применяется к хвосту формата `/page/<alias>/`;
- если `<alias>` совпал с любым regex, отдаётся `404 Not Found`.

Когда использовать:

- для отсечения мусорных добавок ботов: `tel:`, длинные числа, спецсимволы и т.п.

Пример:

```json
"forbidden_alias_patterns": [
  "/^tel:/i",
  "/^\\d{7,}$/",
  "/[\\[\\]&]/"
]
```

Важно:

- это regex, а не обычная строка;
- экранируйте `\\` в JSON;
- слишком широкая маска может убить живые URL.

#### `query_drop_prefixes`

Что это:

- префиксы query-параметров, которые удаляются через `301` редирект.

Как проверяется в коде:

- имя параметра приводится к lowercase;
- если имя начинается с указанного префикса, параметр удаляется.

Когда использовать:

- когда есть целая группа однотипных параметров (`utm_*`, `from_*`, `yclid_*` и т.д.).

Пример:

```json
"query_drop_prefixes": ["utm_", "from_"]
```

Важно:

- это не блокировка страницы, а канонизация URL (`301`);
- параметры удаляются только из адреса, остальные параметры остаются.

#### `query_drop_keys`

Что это:

- точечный список имён query-параметров для удаления через `301`.

Как проверяется в коде:

- имя параметра приводится к lowercase;
- если оно есть в списке, параметр удаляется.

Когда использовать:

- для конкретных ключей, где не нужен префикс;
- для мусорных параметров рекламных/соц/браузерных систем.

Пример:

```json
"query_drop_keys": ["gclid", "fbclid", "search_source", "_ym_debug"]
```

Важно:

- храните ключи в lowercase (так проще читать и поддерживать);
- этот список дополняет `query_drop_prefixes`, а не заменяет его.

#### Приоритет и объединение правил

- сначала используются правила по умолчанию из ядра;
- затем, если есть `projects/<SE_DIR>/urlfilter.json`, его массивы добавляются к дефолтным;
- если основного файла нет, пробуется fallback `projects/<SE_DIR>/cache/urlfilter.json`;
- дубли автоматически убираются;
- полностью выключить дефолт из файла нельзя, можно только расширять.

## 6. Быстрый запуск (пошагово)

1. Открой папку нужного сайта: `projects/<SE_DIR>/`.
2. Создай файл `urlfilter.json`.
3. Вставь пример выше.
4. Сохрани файл.
5. Проверь 5-10 проблемных URL через браузер или `curl`.

## 7. Как проверить, что всё работает

Примеры команд:

```bash
curl -I "https://site.ru/lib/"
curl -I "https://site.ru/page/tel:79520431460/"
curl -I "https://site.ru/page/?utm_source=test"
curl -I "https://site.ru/page/?"
```

Ожидаемо:

- `/lib/` -> `403`
- мусорный хвост -> `404`
- query-мусор -> `301` на чистый URL

## 8. Как добавить новое правило

### Удалять параметр из query

Добавь ключ в `query_drop_keys`, например:

```json
"query_drop_keys": ["gclid", "fbclid", "newparam"]
```

### Удалять группу параметров по префиксу

Добавь префикс в `query_drop_prefixes`, например:

```json
"query_drop_prefixes": ["utm_", "from_"]
```

### Блокировать новый мусорный alias

Добавь regex в `forbidden_alias_patterns`, например:

```json
"forbidden_alias_patterns": ["/^tel:/i", "/^wa\.me/i"]
```

## 9. Частые ошибки

- Неправильный JSON (лишняя запятая, не закрыта скобка).
- Экранирование regex: обратный слэш в JSON нужно писать как `\\`.
- Слишком широкая маска regex, которая режет полезные URL.
- Добавление правил не в тот сайт (`SE_DIR` перепутан).

## 10. Безопасный откат

1. Переименуй `urlfilter.json` в `urlfilter.json.bak`.
2. Проверь проблемный URL.
3. Если причина в правилах, поправь файл и верни имя.

## 11. Что менять в коде, а что нет

В 99% случаев код ядра трогать не нужно.

Обычно меняется только:

- `projects/<SE_DIR>/urlfilter.json`

Код (`seData.class.php`) правится только если нужен новый тип логики, а не новое правило.

## 12. Мини-чеклист перед релизом

- Проверен `200` на обычных страницах.
- Проверен `403` на `/lib/`.
- Проверен `404` на мусорных alias.
- Проверен `301` на query-мусор.
- Нет лишних редирект-цепочек.
- Роботы видят только чистые URL.
